# GRC Learning Lab — Program Overview

**2-week intensive.** By the end you'll have a complete, portfolio-ready GRC program for a fictional cybersecurity startup, built from scratch.

---

## The Big Picture

You're building SecureTech's GRC program the same way a new GRC Analyst would on their first month at a company — starting with governance (the foundation), then risk management (the engine), then compliance (the external requirements), and finally understanding how automation connects it all.

Every artifact you create here is something you'll reference in interviews. The goal isn't just to have files — it's to be able to talk through every decision you made and why.

## Week-by-Week Schedule

### Week 1 — Manual GRC (Building It By Hand)

The point of Week 1 is to deeply understand each component of a GRC program by building it yourself. If you use AI to write your policies and risk entries for you, you're skipping the learning.

| Day | Focus | Deliverables | Est. Time |
|---|---|---|---|
| **Day 1** | Read guides, study worked examples, write first 2 policies | Acceptable Use Policy, Access Control Policy | 6–7 hrs |
| **Day 2** | Finish policies, write all procedures, complete governance overview | 3 remaining policies + 4 procedures + governance overview | 6–7 hrs |
| **Day 3** | Learn risk methodology, build risk register (first 10 risks) | risk_methodology.md, risk_register.md (10 entries) | 5–6 hrs |
| **Day 4** | Finish risk register, complete all treatment plans | risk_register.md complete, risk_treatment_plans.md | 5–6 hrs |
| **Day 5** | Read compliance guide, document NIST CSF + SOC 2 controls | nist_csf_2.md, soc2_criteria.md | 4–5 hrs |
| **Day 6** | Document ISO 27001, begin compliance matrix | iso27001_annex_a.md, compliance_matrix.md (in progress) | 4–5 hrs |
| **Day 7** | Finish compliance matrix, complete gap analysis | compliance_matrix.md complete, gap_analysis.md | 4–5 hrs |

**Week 1 total: ~35–40 hours**

### Week 2 — Automation & Synthesis

Week 2 is about understanding how GRC at scale works. Real GRC programs don't run on manually updated spreadsheets — they use automation tools, GRC platforms, and data pipelines. You'll learn the theory and interview language for this even without writing code.

| Day | Focus | Deliverables | Est. Time |
|---|---|---|---|
| **Day 8** | Understand GRC automation concepts, real-world tools | Read automation theory guide | 2–3 hrs |
| **Day 9** | Learn how the three automation tools work conceptually | Understand risk scorer, compliance mapper, report generator | 1–2 hrs |
| **Day 10** | Write the executive summary | executive_summary.md | 2–3 hrs |

**Week 2 total: ~5–8 hours** (condensed since no coding)

## What "Done" Looks Like

At the end of this program you should be able to:

**In an interview:**
- Walk through any artifact you built and explain every decision
- Describe SecureTech's risk posture from memory (top risks, scores, treatments)
- Explain how NIST CSF, SOC 2, and ISO 27001 overlap using a specific example
- Talk about GRC automation tools and why they matter at scale
- Answer "tell me about a time you built a compliance program from scratch"

**In the portfolio:**
- A GitHub repo with 20+ professional GRC documents
- Evidence you can write policies, assess risk, map frameworks, and synthesize findings for an executive audience

## Tips for Getting the Most Out of This

**Write in your own words.** The templates have prompts — don't just fill them in minimally. The more you write, the more you'll have to say in interviews.

**Use the company profile constantly.** Every policy, risk entry, and control should reference specific details about SecureTech — their tech stack, customer base, compliance deadlines. Generic artifacts don't impress interviewers.

**Think like the CISO.** David Okonkwo (CISO) needs practical, enforceable documents — not academic exercises. Would a real CISO use what you've written?

**Commit to GitHub daily.** Git history shows you did the work. Interviewers sometimes look at commit history to verify a portfolio is real.

**Practice explaining everything out loud.** After completing each module, spend 10 minutes explaining it to yourself as if you were in an interview. This is how you turn documents into stories.
